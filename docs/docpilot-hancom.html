왜 이 프로젝트를 시작했는가
신규 개발 시
  • 기능 구현에 집중하다 보면 명세서 작성은 항상 뒤로 밀림
  • "나중에 정리하자"가 영원히 계속됨
업무 인계 시
  • 명세서 없이 코드만 인계받는 상황
  • Controller를 하나씩 열어보며 엔드포인트 파악
  • 수십 개 Controller → 며칠 소요
문제: 백엔드 개발에서 API 명세서는 필수이지만, 현실적으로 잘 작성되지 않는다.
DocPilot의 해결: GitLab에 코드만 있으면 LLM이 분석하여 Confluence에 API 명세서를 자동 생성. 코드 변경 시 문서도 자동 갱신.

  • 조직 — 프로젝트 전체의 API 현황을 대시보드에서 한눈에 파악.

멀티 플랫폼 지원

프로젝트 등록 시 플랫폼을 선택하면, 플랫폼에 맞는 분석 로직이 자동 적용됩니다.

플랫폼 탐색 패턴 분석 대상 상태
Spring Boot
Java		 *Controller.java in /controller/ @GetMapping, @PostMapping 등 어노테이션 + DTO import 추적 완전 지원
Django
Python		 views.py, viewsets.py, /views/ @api_view, ViewSet, @action + Serializer / urls.py 지원
Express
Node.js		 .js/.ts in /routes/ router.get(), router.post() 등 준비 중
DTO/Serializer 자동 수집: Controller 코드의 import 문을 분석하여 요청/응답 객체의 필드 정보를 자동으로 수집하며, 의존 프로젝트와 공통 라이브러리까지 확장 검색합니다.

Webhook 기반 자동 갱신

개발자가 코드를 Push하면, 변경된 Controller만 감지하여 해당 API 명세서를 자동으로 업데이트합니다.

GitLab Push / Merge Request │ ▼ [Webhook 수신] X-Gitlab-Token 검증 │ ├─ 커밋 메시지에 [skip-doc] 포함? → 건너뜀 ├─ 트리거 브랜치 확인 (main, develop) │ ▼ 즉시 200 OK 반환 (비동기 처리) [변경 파일 분석] │ ├─ 일반 파일 변경 → LLM 분석 → Confluence 업데이트 └─ Controller/View 파일 감지 └─ 플랫폼별 분기 ├─ Spring Boot → ApiSpecInitializationService ├─ Django → DjangoApiSpecService └─ Express → (준비 중)

도입 효과: Before vs After
BEFORE — 수동
  • Controller 코드를 하나씩 열어 엔드포인트 파악
  • 요청/응답 DTO 구조 수동 정리
  • Confluence에 표 형식으로 직접 작성
  • 코드 변경 시 문서와 동기화 안 됨
  • Controller 30개 기준 약 3 ~ 5일 소요
AFTER — DocPilot
  • 프로젝트 등록 1번으로 전체 자동 생성
  • DTO/Serializer 소스 자동 수집 및 반영
  • Confluence에 정형화된 형식으로 자동 작성
  • 코드 Push 시 Webhook으로 자동 갱신
  • Controller 30개 기준 약 30분 소요
