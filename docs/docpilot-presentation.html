1

왜 이 프로젝트를 시작했는가

문제: 백엔드 개발에서 API 명세서는 필수이지만, 현실적으로 잘 작성되지 않는다.
신규 개발 시
  • 기능 구현에 집중하다 보면
    명세서 작성은 항상 뒤로 밀림
  • "나중에 정리하자"가 영원히 계속됨
업무 인계 시
  • 명세서 없이 코드만 인계받는 상황
  • Controller를 하나씩 열어보며
    엔드포인트를 파악해야 함
  • 수십 개 Controller → 며칠 소요
DocPilot의 해결: GitLab에 코드만 있으면, LLM이 분석하여 Confluence에 API 명세서를 자동 생성. 코드 변경 시 문서도 자동 갱신.

기대 효과

  • 개발자 — 코드만 작성하면 명세서가 자동으로 생성됨. 부가 업무 제거.
  • 프론트엔드/QA — 항상 최신 상태의 API 문서를 Confluence에서 확인 가능.
  • 인수인계 — 코드 기반 명세서가 이미 존재하여, 즉시 업무 파악 가능.
  • 조직 — 프로젝트 전체의 API 현황을 대시보드에서 한눈에 파악.

Webhook 기반 자동 갱신

개발자가 코드를 Push하면, 변경된 Controller만 감지하여 해당 API 명세서를 자동으로 업데이트합니다.

GitLab Push / Merge Request │ ▼ [Webhook 수신] X-Gitlab-Token 검증 │ ├─ 커밋 메시지에 [skip-doc] 포함? → 건너뜀 ├─ 트리거 브랜치 확인 (main, develop) │ ▼ 즉시 200 OK 반환 (비동기 처리) [변경 파일 분석] │ ├─ 일반 파일 변경 → 해당 페이지 LLM 분석 → Confluence 업데이트 │ └─ Controller/View 파일 변경 감지 └─ 플랫폼별 분기 ├─ Spring Boot → ApiSpecInitializationService ├─ Django → DjangoApiSpecService └─ Express → (준비 중)

도입 효과: Before vs After

Before (수동)
  • Controller 코드를 하나씩 열어 엔드포인트 파악
  • 요청/응답 DTO 구조 수동 정리
  • Confluence에 표 형식으로 직접 작성
  • 코드 변경 시 문서와 동기화 안 됨
  • Controller 30개 기준 약 3~5일 소요
After (DocPilot)
  • 프로젝트 등록 1번으로 전체 자동 생성
  • DTO/Serializer 소스 자동 수집 및 반영
  • Confluence에 정형화된 형식으로 자동 작성
  • 코드 Push 시 Webhook으로 자동 갱신
  • Controller 30개 기준 약 30분 소요
