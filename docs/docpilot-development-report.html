1. 프로젝트 개요

1.1 배경

소프트웨어 프로젝트에서 문서화는 항상 뒤처지기 마련입니다. 코드는 매일 변경되지만, API 명세서나 서비스 개요 문서는 최초 작성 이후 업데이트가 소홀해지는 경우가 많습니다. DocPilot은 이 문제를 해결하기 위해 탄생한 문서 자동화 에이전트 서버입니다.

1.2 목표

GitLab의 소스 코드 변경을 감지하여, LLM(Large Language Model)이 코드를 분석하고, Confluence에 문서를 자동으로 생성·업데이트하는 end-to-end 파이프라인을 구축하는 것이 핵심 목표입니다.

1.3 기술 스택

2. 아키텍처 설계

2.1 3단계 파이프라인

DocPilot의 핵심 아키텍처는 Source → LLM → Target의 3단계 파이프라인 구조입니다.

[ Source ] [ Core ] [ Target ] GitLab ──┐ ┌──▶ Confluence │ │ Jira ──┼──▶ Webhook/Scheduler ──▶ LLM ──┼──▶ Slack (추후) │ │ (확장) ──┘ └──▶ (확장)

각 단계가 독립적으로 설계되어, 새로운 소스(Jira, GitHub 등)나 출력 대상(Slack, Notion 등)을 추가할 때 기존 코드에 최소한의 영향만 미칩니다.

2.2 서비스 의존성 구조

GitLabWebhookController
  ├─ ConfigLoaderService (설정 조회, 브랜치 필터링)
  └─ DocumentPipelineService (비동기 이벤트 처리)
       ├─ GitLabSourceService (소스 코드 수집)
       ├─ LLMRouter → ClaudeClientService / OpenAIClientService
       ├─ PromptTemplateService (프롬프트 생성)
       ├─ ConfluenceTargetService (페이지 CRUD)
       └─ ApiSpecInitializationService (Controller 변경 시 API 문서 업데이트)

ProjectInitializationService (@Order 1) ── 서버 시작 시 기본 페이지 초기화
ApiSpecInitializationService (@Order 2) ── 서버 시작 시 API 명세서 초기화

2.3 패키지 구조

com.hancom.ai.docpilot.docpilot
├── config/          # 설정 로딩, WebClient, JSON 모델
├── llm/             # LLM 라우팅, Claude/OpenAI 클라이언트, 프롬프트 템플릿
├── source/gitlab/   # GitLab API 연동 (파일 조회, 커밋 분석)
├── target/confluence/ # Confluence REST API (페이지 CRUD, 속성 설정)
└── webhook/         # Webhook 수신, 파이프라인 오케스트레이션, 초기화 서비스

3. 핵심 모듈 구현

3.1 설정 관리 (ConfigLoaderService)

두 개의 JSON 설정 파일을 로드하여 런타임에 활용합니다.

  • config/llm_config.json — LLM 프로바이더 선택. active: true인 항목이 정확히 1개여야 서버가 시작됩니다.
  • config/page_structure_config.json — GitLab 프로젝트와 Confluence 페이지 구조의 매핑을 정의합니다. 브랜치 트리거, glob 패턴, skip 키워드 등을 지원합니다.

AntPathMatcher를 활용하여 변경된 파일과 target_files 패턴을 매칭하고, 어떤 문서 페이지를 업데이트할지 결정합니다.

3.2 LLM 라우팅 (LLMRouter)

설정 파일의 active 플래그에 따라 Claude 또는 OpenAI로 요청을 라우팅합니다. 두 클라이언트 서비스는 동일한 인터페이스(generate(prompt, apiKey, model))를 제공하여 교체가 용이합니다.

3.3 프롬프트 템플릿 (PromptTemplateService)

7개의 사전 정의 프롬프트 템플릿을 관리합니다:

템플릿용도
service_overviewREADME.md → Confluence 형식 변환
api_spec코드 기반 REST API 명세서
controller_analysisController → API 목록 JSON 추출
api_detail / api_detail_with_template개별 API 상세 명세서
db_schemaDB 스키마 문서
changelog변경 이력
weekly_reportJira 기반 주간 보고서

모든 프롬프트는 LLM에게 Confluence Storage Format(XML)으로만 응답하도록 지시합니다. 코드 내용이 8,000자를 초과하면 ...(이하 생략)으로 잘라내어 LLM 컨텍스트 제한을 관리합니다.

3.4 Confluence 연동 (ConfluenceTargetService)

PAT(Bearer) 또는 Basic Auth를 자동 선택하여 인증합니다. 주요 기능:

  • ensureParentPages — 스페이스 홈페이지부터 계층 경로를 탐색하며, 없는 페이지는 자동 생성
  • upsertPageUnderParents — 계층 경로 아래에 문서를 생성 또는 업데이트 (append 모드 지원)
  • setPageWidthNarrow — 페이지 너비를 '좁게'로 설정 (content-appearance-published 속성)

4. API 명세서 자동 생성

4.1 초기화 프로세스

서버 시작 시 ApiSpecInitializationService가 다음 순서로 동작합니다:

  1. Confluence에서 서식 템플릿 페이지 로드 (1회, 캐싱)
  2. GitLab에서 *Controller.java 파일 전체 검색
  3. 기존 Controller 그룹 페이지에서 <!-- controller-path:xxx --> 식별자를 추출하여 이미 처리된 Controller 목록 수집
  4. 미처리 Controller에 대해:
    • LLM으로 Controller 분석 → JSON (API 목록 추출)
    • Controller 그룹 페이지 생성 (Controller 정보 + API 목록 테이블 + 하위 페이지 목차)
    • 각 API 엔드포인트별: 메서드 코드 추출 → LLM으로 상세 명세서 생성 → Confluence 페이지 생성

4.2 메서드 단위 코드 추출

Controller 전체를 LLM에 전달하면 8,000자 제한에 걸릴 수 있습니다. 이를 해결하기 위해 extractMethodCode()가 개별 request handler 메서드만 추출합니다:

  1. 클래스 레벨 @RequestMapping 추출 (base path 정보)
  2. 메서드명으로 해당 메서드 위치를 찾고, 위쪽으로 어노테이션/Javadoc 범위 확인
  3. 중괄호 매칭으로 메서드 끝을 판별

이를 통해 LLM에 전달되는 코드량을 대폭 줄이고, 더 정확한 API 명세서를 생성할 수 있었습니다.

4.3 API 상세 명세서 구성

각 API 페이지는 다음 섹션을 포함합니다:

섹션내용
개요API의 목적과 기능 설명
Request URL[HTTP메서드] {domain}/전체/경로 형식
Request Parameter파라미터명, 타입, 필수여부, 설명 (테이블)

5. 서식 템플릿 시스템

5.1 문제

LLM이 생성하는 Confluence 페이지의 서식이 매번 달랐습니다. 테이블 스타일, 섹션 순서, 페이지 너비 등이 일관되지 않아 문서 품질이 들쭉날쭉했습니다.

5.2 해결: 서식 예시 페이지 기반 생성

Confluence에 "API 명세서 예시"라는 서식 템플릿 페이지를 수동으로 작성하고, 서버 시작 시 해당 페이지의 Storage Format XML을 한 번 읽어 캐싱합니다.

서버 시작
  ↓
Confluence "API 명세서 예시" 페이지 조회 (body.storage)
  ↓
cachedTemplateXml에 캐싱 (1회)
  ↓
각 API 페이지 생성 시 프롬프트에 XML 원문 주입
  ↓
LLM이 동일한 서식으로 문서 생성

5.3 Truncation 분리

초기 구현에서는 서식 템플릿 XML과 API 코드가 합쳐진 상태로 8,000자 truncation이 적용되어 서식이 잘리는 문제가 있었습니다. 이를 해결하기 위해 getPromptWithTemplate() 메서드를 분리하여:

  • 서식 템플릿 XML → truncation 없이 온전히 보존
  • API 코드 → 8,000자 truncation 적용

5.4 페이지 너비 강제 설정

LLM이 서식 템플릿의 너비 설정(좁게)을 완벽히 따르지 못하는 한계가 있었습니다. 이를 보완하기 위해 페이지 생성 후 Confluence REST API로 content-appearance-published 속성을 fixed(좁게)로 직접 설정합니다.

6. 중복 방지 메커니즘

6.1 문제

LLM이 같은 Controller에 대해 매번 다른 한국어 이름을 생성했습니다:

실행 차수ChatController.java
1차[한컴 챗봇 백엔드] 챗봇
2차[한컴 챗봇 백엔드] 챗봇관리
3차[한컴 챗봇 백엔드] 챗봇대화
4차[한컴 챗봇 백엔드] 챗봇채팅

제목 기반 중복 체크로는 동일 Controller의 페이지가 중복 생성되는 문제를 막을 수 없었습니다.

6.2 해결: Controller 파일 경로 기반 식별

Controller 그룹 페이지 본문 상단에 고유 식별자를 삽입합니다:

<!-- controller-path:src/main/java/.../ChatController.java -->

서버 시작 시 기존 페이지들에서 이 식별자를 추출하여 이미 처리된 Controller Set을 구성합니다. Controller 파일 경로는 LLM 출력에 의존하지 않는 결정적(deterministic) 값이므로, 서버를 몇 번 재시작하든 동일 Controller에 대해 중복 페이지가 생성되지 않습니다.

핵심 이점: LLM 호출 전에 중복을 판별하므로, 이미 처리된 Controller에 대해서는 불필요한 LLM API 비용이 발생하지 않습니다.

7. Webhook 기반 실시간 업데이트

7.1 Webhook 수신 흐름

POST /webhook/gitlab 엔드포인트가 GitLab Push/MR 이벤트를 수신합니다:

  1. X-Gitlab-Token 헤더로 인증 검증
  2. 이벤트 유형 판별 (Push Hook → "push", Merge Request Hook → "merge_request")
  3. 커밋 메시지에서 skip 키워드([skip-doc]) 확인
  4. 브랜치 트리거/무시 패턴 매칭
  5. @Async로 비동기 파이프라인 실행 → 즉시 200 OK 반환

7.2 Controller 변경 감지

DocumentPipelineService는 일반 페이지 처리 후 변경 파일 목록에서 Controller 파일을 감지합니다:

changedFiles.stream()
    .filter(f -> f.contains("/controller/") && f.endsWith("Controller.java"))
    .filter(f -> !f.contains("ErrorHandler") && !f.contains("CustomError"))

변경된 Controller가 있으면 ApiSpecInitializationService.processControllerUpdate()를 호출하여 해당 API 명세서 페이지를 업데이트합니다. 이때 forceUpdate=true로 동작하여 기존 페이지 내용이 최신 코드 기반으로 갱신됩니다.

9.3 얻은 교훈

  • LLM 출력은 비결정적이다 — 동일 입력에도 다른 결과가 나옵니다. 식별자나 키값은 LLM 출력이 아닌 결정적 데이터(파일 경로, ID 등)에 의존해야 합니다.
  • 프롬프트만으로는 한계가 있다 — 서식의 세밀한 부분(너비, 특정 CSS 속성)은 프롬프트로 100% 제어하기 어렵습니다. API를 통한 후처리로 보완하는 것이 실용적입니다.
  • 점진적 처리가 현실적이다 — 32개 Controller를 한 번에 처리하면 60분 이상 소요됩니다. 서버 시작마다 일부씩 처리하는 점진적 접근이 운영 환경에 적합합니다.
  • 테스트가 리팩토링의 안전망이다 — 58개의 기존 테스트가 있어, 구조 변경 시마다 빠르게 회귀를 확인할 수 있었습니다.
